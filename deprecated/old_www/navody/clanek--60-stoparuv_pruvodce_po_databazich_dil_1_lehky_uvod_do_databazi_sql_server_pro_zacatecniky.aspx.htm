Lehký úvod do databází - SQL Server pro začátečníky

[Zpět na kategorii]

Datum: 15. 1. 2008 2:57       Autor: Tomáš Jecha       Zobrazeno: 2613x

Kategorie: Databáze

Témata: SQL

Seriál: Stopařův průvodce po databázích - Díl 1.

První díl nového seriálu je věnován absolutním začátečníkům v této oblasti. Povíme si něco o SQL Serveru 2005, naučíme se základní práci s SQL Server Management Studiem, vytvoříme si databázi a probereme zakládání tabulek...

Byla nebyla početná skupina programátorů, kteří alespoň jednou za život využili databázi. Ale nemalá část z nich dokáže pracovat jen s velmi omezenou škálou funkcionality, co nabízí. Často ani neví, že mohou použít nějaké uložené procedury nebo transakce. Když se pak někdo takový pokusí sestavit větší systém založený na databázích, v práci se nakonec zamotá. Celé to pak může být děravé a nemalou část času tráví ošetřováním nejrůznějších problémů. Jenže při správném návrhu můžete prakticky úplně zamezit, aby se do databáze uložilo něco špatně. Proto jsem se rozhodl napsat tento seriál o Microsoft SQL Serveru. Rád bych narozdíl od jiných článků zašel do hloubky a popsal i složitější vychytávky, které můžou ve výsledku o dost usnadnit práci. I to jest důvodem, proč budu popisovat problémy spíš z pohledu databázového návrháře než programátora. Navíc nechci vysvětlovat věci jen teoreticky, budu uvádět příklady, kde se popisované funkce dají využít a popřípadě, zda se dají nahradit jinou funkcí. Dotazy rád odpovím, pokud na to budou mé schopnosti stačit.

Databáze obecně

Tento kus článku je pro opravdové začátečníky, takže kdo už s nějakou databází někdy pracoval, může ho s klidem v duši přeskočit.

Databáze není nic jiného než uložiště na data. I jednoduchý soubor s odřádkovaným textem je vlastně databáze. A kus kódu, který ho parsuje je vlastně databázový systém. U menších projektů můžete občas váhat zda použít řešení podobné textovému souboru nebo velký databázový server. Patrně tušíte, že zde bude rozdíl v robustnosti výsledného řešení. Přeci jenom SQL Server budovaný početnou skupinou profesionálů několik dlouhých let v porovnání s vlastním systémem ukládání, který si napíšete za 5 minut bude nabízet o něco komplexnější služby. Tím nechci říkat, že nasadit těžký server musí být vždy lepší, ale pokud budujete něco většího, určitě se vyplatí zvážit jeho výhody:

  • řešení několikanásobného přístupu - zkuste si zapsat 5x vedle sebe do jednoho souboru a máte skoro jistotu, že se program sesype. V databázích můžete ty kolize při dobrém návrhu velmi elegantně řešit.
  • několik možných druhů přístupů - přistupujte si do databáze ze sítě, či z internetu přes TCP/IP nebo z lokálního stroje přes sdílenou paměť - je čistě na vás, co vám zrovna vyhovuje
  • bezpečnostní politika - disponují silnou bezpečnostní politikou, která umožní dobře aplikace zabezpečit a omezit přístup škůdcům
  • zvládá větší objemy dat - databáze jsou navržené aby zvládaly procházet a modifikovat i velké množství dat ve velmi krátkém čase (psát ručně indexování, stránkování a binárně optimalizované ukládání je pro běžného smrtelníka velmi optížné)
  • SQL jazyk - dává nám možnost snadno řešit nejrůznější třídění, prohledávání, spojování atp.

Když to všechno sečteme, nabízí nám SQL Server robustní řešení ukládání dat. I proto jsou dnes databáze nezastupitelným prvkem při budování většiny aplikací a to jak desktopovým, internetových i mobilních.

Ale jak ukládat data? Databáze může obsahovat libovolný počet tabulek, které slouží právě k ukládání dat (řádků - rows). Tyto řádky musí být ve formátu, který určujou předem definované sloupce (columns) tabulky. Takže pokud chceme ukládat například uživatele, vytvoříme do databáze tabulku a přidáme jí 3 sloupce: uživatelské jméno, plné jméno a heslo. Sloupce musí mít unikátní jména, takže například: userName, fullName a password. Když potom vkládáme uživatele, přiřadíme sloupcům hodnoty (například: userName='tomas',fullName='Tomáš Jecha' a password='tajne heslo'). Samozřejmě je řada dalších nastavení a úprav tabulky, které by bylo zapotřebí nastavit (třeba znemožnit duplicitní názvy sloupce userName), ale těmi se budeme zabívat později v článku. Navíc je dobré vytvářet unikátní identifikátory řádků, kterými můžete příslušný záznam identifikovat. V našem případě by to byl právě sloupec s uživatelským jménem. Často se ale používá číslo (můžeme nastavit, že se budou záznamy automaticky inkrementovat) - tzn. že každý záznam má své unikátní číslo, tzv. ID. Tyto sloupce se nazívají klíčové. Jejich výhodou není jen snadná unikátní identifikace, ale i možnost provázat více tabulek dohromady - o tom si povíme ale později.

Na čem si budeme příklady ukazovat

Doporučuji si stáhnout SQL Server 2005 Express, který je zdarma i pro komereční využití. Jako administrační rozhraní použijeme velmi kvalitní SQL Server Management Studio, které si v tomto prvním díle také popíšeme. Obě dvě aplikace můžete získat v tomto balíčku: Microsoft SQL Server 2005 Express Edition with Advanced Services

Pokud máte chuť nebo potřebu zkoušené příklady i přímo nějakým způsobem hned implementovat, můžete si stáhnout Visual Basic .NET Express Edition nebo kterýkoliv jiný programovací jazyk, který se do MSSQL Serveru dokáže připojit. Jak na to, jsem popisoval už v jiném článku:
Komunikace s MSSQL databází (VB.NET, C#)

Pár vět k samotnému SQL Serveru 2005

Po instalaci SQL Serveru se nám do nabídky start přidala složka Microsoft SQL Server 2005. V ní nalezneme především SQL Server Management Studio Express (nástroj na správu databáze). Dále nás pak bude zajímat SQL Server Configuration Manager (konfigurace samotných procesů nainstalovaných databází) v podsložce Configuration Tools.

Než se ale vrhneme na samotnou konfiguraci, musíme si vysvětlit několik základních pojmů:

  • Databázový server (myšleno jako software) - zpřístupňuje soubor databází uživatelům, programárům a aplikacím podle bezpečnostních nastavení
  • Databáze - soubor tabulek, uživatelů, oprávnění, databázových pravidel atp. - databází může v jednom databázovém serveru existovat několik
  • Tabulka - datové uložiště obsahující sloupce definující formát ukládaných dat - tabulky mohou být provázané a mohou obsahovat nejrůznější pravidla

Je důležité vědět, že s nainstalováním SQL Serveru Express edice se nám do počítače zavede jeden databázový server pojmenovaný SQLEXPRESS. Ten běží jako samostatný proces systému Windows. Když si spustíme správu procesů nebo SQL Server Configuration Manager, tak můžeme tento proces vidět:

Při poklepání uvidíme standardní nastavení procesů - způsob spuštění (defaultně nastaven automatický po startu Windows), účet procesu, kontrola spuštění atp. To není pravděpodobně nic, co teď potřebujeme měnit. Zajímavější je ale druhá položka nastavení. A to SQL Server 2005 Network Configuration. Zde se povolují pro databázové servery způsoby komunikací a jejich nastavení - například komunikační porty pro TCP/IP atp. Myslím, že je dobré o těchto nastaveních vědět. Osobně je ale nedoporučuji měnit do té doby, než to budete opravdu potřebovat.

SQL Server Configuration Manager totiž nezasahuje přímo do konkrétní databáze - spravuje jen proces databázového serveru, který nám pak tvoří jakousi bránu do databáze pro naše aplikace. Z toho důvodu se používá většinou jen při prvotním nastavením. Ale jak už jsem psal, nás se to zatím stejně netýka, prvotní nastavení nám vyhovuje.

O mnoho zajímavější je samostatný program SQL Server Management Studio. Ten se chová stejným způsobem jako obyčejná aplikace využívajíc nějakou databázi. Komunikuje jen SQL příkazy (přesněji Transact-SQL, což je implementace SQL jazyku od Microsoftu) a slouží už ke konkrétní práci s databází. A protože pokrývá prakticky vše, co je možné s SQL Serverem dělat, budeme ho používat na většinu ukázek v tomto seriálu. Přihlášení na server

Po spuštění SSMS na nás vyskočí okno přihlášení:

  • První položka je zešedlá (Server type). Ta nás zatím nemusí trápit - jedná se totiž o výběr typu databáze. Konkrétně se tu objeví po doinstalování například SQL Server Compact Edition pro mobilní zařízení.
  • Server name - zde vepisujete do jakého databázového serveru se připojit. Jak už jsem zmínil, na jednom PC může být více procesů a z toho důvodů se uvádí nejdříve jméno počítače a pak jméno procesu databázového serveru: jméno_počítače\jméno_procesu_databázového_serveru. Možná vás proto mate ta tečka, kterou začíná adresa - je to zástupný symbol pro lokální stroj, takže by fungovalo i například localhost\SQLEXPRESS.
  • Authentication - k přihlašování do SQL Serveru je možné obecně použít 2 zcela odlišné způsoby: Windows Authentication (přihlášení pomocí účtů ve Windows) nebo SQL Server Authentication (přihlašování jménem a heslem definovaném v SQL Serveru). Přihlašování a bezpečnost budu probírat v jednom z dalších článků. Zatím nám postačí Windows Authentication (použije se aktuální uživatelský účet, kterým jsme přihlášeni - proto je potřeba, aby byl administrátorský a my tak měli plný přístup do databáze).
  • Další volby pomocí tlačítka Options >>. Zde dostáváte možnost si zvolit konkrétní databázi do které se po přihlášení připojíte (jak už jsem psal, v jednom serveru může být více databází), ale to se používá většinou jen při spouštění skriptů ze souborů, které v sobě nemají uvedenou databázi proti které se spustí, takže se nás to opět netýká - my si ji můžeme pohodlně vybrat po přihlášení. Další nastavení se pak týka připojení a komunikace. Zase jsou to věci pro nás celkem nepodstatné a základní nastavení je zatím zcela vyhovující.

Teď se přihlásíme. Máme vybraný server .\SQLEXPRESS a zvolenou autentifikaci přes účty Windows. Klepneme na Connect. Dostaneme se do hlavního okna SSMS.

Hlavní okno SQL Server Management Studia

Hned po spuštění uvidíme dvě hlavní části oken. Procházení objektů (Object Explorer) a jeden list souhrnu objektů na serveru (Summary). Poslední okno vlastností (Properties) se zobrazí, až když bude potřeba (když bude co editovat).

Nejdůležitejší pro naši navigaci bude okno Server exploreru. Když rozklepnete Databases, uvidíme všechny databáze, která na serveru jsou. V každé z nich pak máme možnost prohlížet tabulky (Tables), grafické databázové diagramy (Database Diagrams), pohledy na tabulky (Views) a řadu dalších. SSMS nám dává moc tyto objekty upravovat nebo přidávat další. Uděláme to otevřením kontextového menu pravým tlačítkem myši nad příslušným objektem. Takže pro přidání nové databáze zobrazíme kontextové menu položky Databases pod kterým se právě skrývají všechny ostatní umístěné na serveru:

Důležité je si uvědomit, že pokud chceme provést nějakou akci, budeme jí hledat v kontextovém menu odpovídajícího objektu. Uvedu ještě jeden příklad. Chceme do právě vytvořené databáze (pojmenované třeba eshop) vložit tabulku. Rozklepneme si Databases/eshop/Tables a zde v kontextovém menu zvolíme New Table:

Díky tomu se otevře editace nové tabulky do hlavního panelu, kde jí můžeme pohodlně nastavit. Po uložení se tabulka objeví v Object Exploreru tam, kde ji budemem čekat, v Databases/eshop/Tables/jmenoTabulky. I zde platí, že je možné ji dále otevřít a podívat se třeba na sloupce tabulky. A když na ní teď klepneme opět pravým tlačítkem, dostaneme na výber modifikaci tabulky, smazání a otevření pro editaci a mazání záznamů.

Posledními několika odstavci jsem chtěl ukázat alespoň trochu jak je SSMS navrženo. Důležité je si pamatovat, že pro navigaci po serveru slouží Object Explorer, který nabízí prakticky všechny úkony s databází. Editace, úpravy a procházení se pak odehrávají v hlavním okně a oknu vlastností. Díky záložkám můžete mít navíc otevřených několik tabulek nebo dotazů najednou.

Doufám, že jsem vás tím moc nezamotal a i když vám ještě není něco úplně jasné, nezoufejte, vše se dovysvětlí v průběhu seriálu.

Než vytvoření databáze potvrdíte, všimněte si několika věci. První z nich je tlačítko Script. Jak už jsem říkal, SSMS komunikuje s databázovým serverem jen pomocí SQL příkazů a tak si tímto tlačítkem můžete výsledný kód zobrazit. Proto po klepnutí obdržíte kód začínající příkazem na vytvoření nové databáze CREATE DATABASE [eshop]... a končící docela obstojnou hromadou nastavení, které najdete v záložce Options a Filegroups. Zatím ale jednotlivé nastavení vysvětlovat nebudu a smíříme se s tím výchozím. Klepneme na OK.

Datové typy v SQL Serveru 2005

Ještě než začneme tvořit jednotlivé tabulky, popíšeme si datové typy, co SQL Server 2005 nabízí. Rozdělují se do několika základních kategorií:

Exact numerics - přesné číselné typy

  1. BIGINT - 8 bytů (rozsah -2^63 až 2^63-1)
  2. INT - 4 byty (rozsah -2^31 až 2^31-1)
  3. SMALLINT - 2 byty (rozsah -32,768 až 32,767)
  4. TINYINT - 1 byte (rozsah 0-255)
  5. BIT - 1 bit (rozsah 0-1)
  6. DECIMAL(p,s) - číslo s desetinnou čárkou (p je maximum cifer celkem a s je maximální počet desetiných mít, takže 0<=s<=p)
  7. NUMERIC(p,s) - to samé co decimal
  8. MONEY - desetinné číslo na desetitisíciny - 8 bytů (-922,337,203,685,477.5808 to 922,337,203,685,477.5807)
  9. SMALLMONEY - také desetinné číslo na desetitisíciny - 4 byty (- 214,748.3648 to 214,748.3647)

Approximate numerics - přibližné číselné typy

  1. FLOAT(n) - číslo s plovoucí desetinnou čárkou, kde n (1-53) určuje počet bitů - přesnost čísla je přibližná a přímo ovlivněná celkovým počtem cifer - celkem vždy zabírá 4 nebo 8 bytů
  2. REAL - je to samé jako FLOAT(24)

Date and time - datum a čas

  1. DATETIME - rozsah od 1. ledna 1753 do 31. prosince 9999 (přesnost 3.33 milisekund)
  2. SMALLDATETIME - rozsah od 1. ledna 1900 do 6. června 2079 (přesnost 1 minuta)

Character strings - běžné textové řetězce (1 byte na znak)

  1. CHAR(n) - řetězec s pevnou délkou n znaků (1 až 8000) - nepoužité znaky budou mít nulovou hodnotu
  2. VARCHAR(n) - řetězec s variabilní délkou, kde n udává maximální délku (opět 1 až 8000) - pokud se místo n použije klíčové slovo MAX - tedy VARCHAR(MAX) - maximální délka je v tomto případě 2^31-1
  3. TEXT - místo tohoto datového typu se doporučuje používat VARCHAR(MAX)

Unicode character strings - unicode textové řetězce (2 byty na znak)

  1. NCHAR(n) - viz CHAR(i), jen zabírá 2x víc místa v paměti (unicode) a dokáže tak lépe zpracovat například diakritiku nebo obecně nestandardní znaky
  2. NVARCHAR(n)- viz VARCHAR(i), jen zabírá 2x víc místa v paměti (unicode) a dokáže tak lépe zpracovat například diakritiku nebo obecně nestandardní znaky
  3. NTEXT- místo tohoto datového typu se doporučuje používat NVARCHAR(MAX)

Binary strings - binární formáty

  1. BINARY(n) - datový typ s pevnou délkou n bytů (rozsah 1 až 8000) pro ukládání binárních dat (souborů, obrázků, atp.)
  2. VARBINARY(n) - datový typ s variabilní délkou, kde n udává maximální kapacitu v bytech (rozsah opět 1 až 8000) - stejně jako u VARCHAR lze použít klíčové slovo MAX, které zaručí nastavení maximálního počtu bytů na 2^31-1
  3. IMAGE - místo tohoto datového typu se doporučuje používat VARBINARY(MAX)

Other data types - ostatní datové typy

  1. CURSOR - budu popisovat později, nelze využít jako datový typ sloupce tabulky
  2. TABLE - slouží k dočasnému uložení datové tabulky - opět nelze použít jako typ sloupce
  3. SQL_VARIANT - dokáže obsahovat různé datové typy (INT, DECIMAL, CHAR, BINARY, NCHAR) - použitý formát se určuje automaticky podle vstupu
  4. TIMESTAMP - i přes název neukládá tento 8mi bytový binární typ datum ani čas - sloupec typu TIMESTAMP může být v tabulce jen jeden a automaticky se inkrementuje při každém vložení nebo změně řádku - slouží tím pádem pro verzování v rámci jedné tabulky
  5. UNIQUEIDENTIFIER - jedná se o 16ti bytový unikátní identifikátor ve tvaru xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx, kde každý znak je hexadecimální znak (0-9 nebo A-F) - pokud to neurčíte jinak, může se v rámci tabulky opakovat - nový a náhodný vygenerujete funkcí NEWID(), což je možné zapsat jako defaultní hodnotu
  6. XML - tento datový typ dokáže ukládat XML struktury  a to buď naprosto volně nebo podle schématu - dále je možné uložit buď celý XML dokument i s hlavičkou nebo jen jeho část (větev)

Volba správného datového typu a popřípadě jeho velikosti je základ dobře navržené databáze. Není to ani nic složitého, jen se musí člověk trochu zamyslet. Často se totiž setkávám s problémem, že programátoři znají jen pár datových typů a navíc je používají špatně. Nikdo neříká, že se musíte naučit velikosti jednotlivých typů v bytech (nejsme přece ve škole, aby jsme cpali do hlavy věci násilím), ale stačí, když budete mít přehled někdě po ruce a on se vám pomalu dostane do krve.

První tabulka - seznam výrobců

Dočetli jste se až sem? Výborně. Myslím, že teď už jsme dostatečně způsobilí na vytvoření prvních tabulek pro náš ukázkový eshop. První navrhneme tabulku, ve které budou všichni výrobci a dodavatelé zboží. Bude se jmenovat Companies. Pro začátek bude obsahovat tyto sloupce:

  • CompanyID - unikátní celočíselný identifikátor firmy (bude se přidělovat automaticky při vložení řádku) - použijeme INT
  • CompanyName - jméno firmy - použijeme NVARCHAR(100)
  • CompanyDescription - popis (nemusí být zadaný) - použijeme NVARCHAR(500)

Máme založenou prázdnou databázi a můžeme přidat novou tabulku (v object exploreru vybereme složku Tables a v kontextovém menu zvolíme New Table). Objeví se nám okno pro editaci sloupců nové tabulky:

Nejdůležitějším prvkem je editace sloupců - první je název sloupce, druhý jeho datový typ a poslední je výběr, zda umožníme nechat sloupec prázdný - tedy aby měl hodnotu NULL.

V pravém sloupci vidíme jméno tabulky (Name - pokud ho necháme automatické, tak se nás SSMS při ukládání zeptá na nové pojmenování) a popis (Description - libovolný textový, jen pro orientační účely - např. V této tabulce je seznam dodavatelů a výrobců.). Ostatní nás nebudou zajímat.

Ve vlastnostech sloupce nalezneme další důležité nastavování:

  • Name, Data Type a Allow Nulls - jméno sloupce, datový typ a povolení vložit prázdnou hodnotu - to jsme už nastavili v horním seznamu sloupců
  • Default Value or Binging - tady můžete zadat defaultní hodnotu pro konkrétní sloupec. Zatím to nevyužijeme, ale později se to bude hodit například pro automatické vyplňování datumu atp.
  • Collation - využitelné jen pro textové sloupce - udávají se tu pravidla pro třídění řetezců. Opět zatím nebudeme využívat.
  • Computed Column Specification - tady máme možnost nechávat hodnotu sloupce automaticky vypočítávat - to se bude hodit pro počítání počtu výrobků, které dodává určitá firma. Na to si ale ještě opět počkáme.
  • Description - opět popis, tentokrát v rámci sloupce, ne celé tabulky.
  • Identity Specification - automatická inkrementace - po rozbalení této položky lze zapnout a nastavit krok inkrementace.

Do popisu ostatních sloupců bych se zatím raději nepouštěl, bylo by to zbytečné, jde povětšinou jen o informační položky. Další neméně důležité nastavení nám skrývá každý sloupec a jeho kontextové menu:

  • Set Primary Key - vytvoří ze sloupce klíčový sloupec

! Něco málo ke klíčům - klíčem může být maximálně jeden sloupec v tabulce s datovým typem o velikosti maximálně 900 bytů, který nepovoluje prázdné (NULL) hodnoty a je v rámci tabulky unikátním. S klíčem se zároveň vytváří indexování - to velmi urychluje vyhledávání v záznamech právě podle něj. Indexovat můžete však nechat více sloupců.

  • Insert Column - nový sloupec
  • Delete column - smaže sloupec
  • Relationships - provázanosti sloupců a tabulek
  • Indexes/Keys - seznam klíčů a indexů v tabulce
  • Fulltext Index - seznam fulltextových indexů
  • XML Indexes - indexy podle XML dokumentu
  • Check Constraints - omezení pro tabulku
  • Generate Change Script - vygeneruje příkaz v jazyce TSQL, který se bude odesílat na server (něco jako tlačítko Script při vytváření databáze) - takže při vytváření tabulky bude začínat CREATE TABLE [jmenoTabulky] ...

Náš postup - tabulka výrobců ve zkratce

Novou tabulku pojmenujeme Companies a přidám tři sloupce. První CompanyID datového typu INT s nastavenou automatickou inkrementací. Zároveň tento sloupec přes kontextové menu nastavte jako primární klíč (ukáže se tam žlutá ikonka klíče). Druhý sloupec pojmenujeme CompanyName a jeho datový typ bude NVARCHAR(100) a nebude povolovat prázdnou hodnotu (vypnout Allow nulls). Poslední sloupec bude CompanyDescription. Aby se tam textu vešlo víc, tak to bude NVARCHAR(500) a bude povolovat prázdnou hodnotu. Pak klepněme na uložit (Ctrl+S). Tabulka by se měla objevit v seznamu v naší databázi. Když jí teď otevřete přes příkaz Open Table z kontextového menu právě vložené tabulky, ukáže se její (zatím prázdný) datový obsah:

Jen si nespleťte Open Table a Modify. Open Table slouží ke čtení dat a zápisu řádků do tabulky a Modify nás naopak přesune do editace jednotlivých sloupců (podobně jako při vkládání nové tabulky).
